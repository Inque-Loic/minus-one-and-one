项目已经从“可运行原型”推进到“可公开下载 Demo”：

当前最主要的短板不是功能可用性，而是第一次游玩时的理解成本。

## 3. 保留范围与取消范围

### 保留

- 阵营胜负。
- 猜分结果摘要。
- 简洁 UI。
- CSV 参数读取和测试模式。
- Windows Demo 发布。

### 取消

- 完整终局复盘系统。
- 逐回合时间线回放。
- 复杂接触链、谎言链或推理图谱。

这些取消项不影响当前 Demo 目标，反而有利于保持界面和开发范围简单。

## 4. 下一阶段推荐顺序

### P0：新手引导补强

目标：让第一次打开游戏的人能在 30 秒内理解：

- 自己是什么阵营。
- 本局目标是什么。
- 三个主按钮分别影响什么。
- 为什么要猜分或观察别人。

实现状态：

- 已在开始游戏前增加一页“快速规则”。
- 身份页保留现有信息，但压缩为更清晰的短句。
- 不增加复杂教程流程，不做多层弹窗。

### P1：发布页素材

目标：让 Itch 和 GitHub 页面看起来可信、可理解。

建议补充：

- 主菜单截图。
- 身份说明截图。
- 局内行动截图。
- 测试模式 1000 局截图。
- 1 段极短玩法说明。

### P1：平衡微调

当前 1000 局结果：

- 一阵营胜：58.0%
- 负一胜：35.5%
- 平局：6.5%

这可以接受，但后续可以尝试：

- 轻微提高负一成功收益。
- 调整 AI 拒绝和接触倾向。
- 调整猜分参与率或猜分奖励。

### P2：配置表说明

目标：让 Portfolio 文件夹能解释项目如何调参。

建议增加：

- `game_rules.csv` 字段说明。
- `ai_tuning.csv` 字段说明。
- 当前参数为什么选这个版本。

## 6. 发布维护流程

每次准备发布新版本时：

1. 手动完整玩 1 局。
2. 批量测试至少 1000 局。
3. 检查中文字体和按钮点击。
4. 更新 `Docs/Portfolio/BalanceTestReport.md`。
5. 更新 `Docs/Portfolio/KnownIssues.md`。
6. 生成 Windows Build。
7. 打包 zip。
8. 上传 Itch。
9. 创建或更新 GitHub Release。

## 8. 判断标准

下一阶段不要追求“功能更多”，优先追求：

- 玩家一眼知道该点什么。
- 玩家知道自己的行为为什么重要。
- 发布页能清楚说明这是一个可玩的社交推理原型。
- 代码和文档能支持继续调参。
