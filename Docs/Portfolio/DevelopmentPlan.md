# 从原型到可发布 Demo 的开发计划

## 1. 当前基线

| 维度 | 状态 |
| --- | --- |
| Unity 版本 | 6000.3.10f1 |
| 运行形态 | 单机人机原型 |
| 主场景 | `Assets/Scenes/SampleScene.unity` |
| 核心脚本 | `GameManager.cs`、`UIManager.cs`、`Player.cs` |
| 当前优势 | 完整状态机、AI 流程、批量测试入口已经存在。 |
| 当前短板 | 文案乱码风险、规则硬编码、UI 复盘弱、缺少正式测试与构建流程。 |

## 2. Demo 目标定义

Demo 目标不是做完整商业版，而是做一个可以给玩家、老师、招聘方或合作方演示的系统策划样片：

1. 5 分钟内讲清楚规则。
2. 15 分钟内完成一局。
3. 局后能看懂谁做了什么、为什么赢或输。
4. 策划能通过配置和批量测试解释平衡变化。
5. Windows 包可独立运行，不依赖 Unity Editor。

## 3. 里程碑计划

### M0：工程整理与风险清零，0.5-1 天

| 任务 | 交付物 | 验收 |
| --- | --- | --- |
| 建立文档目录 | `Docs/Portfolio` | PRD、计划、竞品、配置表齐全。 |
| 统一编码检查 | 脚本和场景中文文案清单 | Unity 显示与源码读取均不乱码。 |
| 建立版本记录 | `CHANGELOG.md` 或文档内版本表 | 每次规则改动可追踪。 |
| 明确 Demo 平台 | Windows x64 | Build Settings 正确。 |

### M1：规则数据化，1-2 天

| 任务 | 交付物 | 验收 |
| --- | --- | --- |
| 抽离全局规则参数 | `GameRulesConfig` 或 JSON/CSV 加载器 | 总人数、负一数、回合数、初始分可调。 |
| 抽离 AI 参数 | `AITuningConfig` | 怀疑度增减、拒绝概率、跳过概率可调。 |
| 增加配置校验 | Editor/Runtime 校验方法 | 负一人数小于总人数、回合数大于 1 等非法配置会报错。 |
| 保留默认配置 | `DefaultDemoConfig` | 不改 Inspector 也能直接开始游戏。 |

### M2：可解释日志与复盘，2-3 天

| 任务 | 交付物 | 验收 |
| --- | --- | --- |
| 建立事件日志模型 | `GameEvent` 数据结构 | 记录回合、阶段、玩家、动作、分数变化。 |
| 接触日志 | 行动/接受/拒绝/放弃事件 | 终局可回看完整接触链。 |
| 讨论日志 | 发言类型、目标、声明分数 | 可追踪谁影响了谁的怀疑度。 |
| 猜分日志 | 猜测值、正确数、奖惩 | 终局解释猜分影响。 |
| 复盘 UI | 结算页新增“本局记录”区域 | 玩家能看懂胜负原因。 |

### M3：新手体验与 UI 打磨，2-4 天

| 任务 | 交付物 | 验收 |
| --- | --- | --- |
| 身份页重写 | 阵营目标、核心规则、同伴提示 | 玩家不会误解自己目标。 |
| 行动可用性提示 | 不可选目标说明 | 禁用按钮有明确原因或视觉状态。 |
| 讨论页布局整理 | 发言记录、按钮、目标列表分区 | 1920x1080、1366x768 无遮挡。 |
| 猜分页优化 | 默认输入、跳过、提交、错误提示 | 空输入/非法输入不会导致异常。 |
| 统一视觉反馈 | 接触成功、拒绝、坏人行动、猜分结果 | 每个阶段结束有明确反馈。 |

### M4：AI 与平衡测试，2-3 天

| 任务 | 交付物 | 验收 |
| --- | --- | --- |
| AI 决策原因记录 | 日志字段：选择原因、怀疑度、随机阈值 | 复盘时能解释 AI 为什么这么做。 |
| 批量测试报告 | 控制台或 UI 输出统计 | 100/1000 局能稳定跑完。 |
| 参数扫描 | 多组回合数/负一人数/初始分组合 | 产出推荐 Demo 参数。 |
| 平衡结论 | 文档新增测试结果 | 明确当前版本偏向哪一阵营。 |

### M5：Demo 构建与验收，1-2 天

| 任务 | 交付物 | 验收 |
| --- | --- | --- |
| Windows 构建 | `Builds/Demo/` | 双击可运行完整一局。 |
| 冒烟测试 | 手动测试清单 | 主菜单、正常局、测试局、结算均通过。 |
| 已知问题清单 | `KnownIssues.md` | 非阻塞问题明确记录。 |
| 作品集整理 | 截图、GIF、设计说明 | 可用于简历/面试展示。 |

## 4. 推荐开发顺序

1. 修文案/编码和 UI 文本，因为这是 Demo 第一印象。
2. 抽配置，把硬编码参数变成可调资产。
3. 加事件日志，让系统从“能跑”变成“能解释”。
4. 做终局复盘，把玩法价值展示出来。
5. 跑批量测试，用数据反推默认参数。
6. 最后做构建、截图和作品集包装。

## 5. 任务优先级

| 优先级 | 任务 | 原因 |
| --- | --- | --- |
| P0 | 中文文案不乱码 | 任何 Demo 展示都会先被这件事影响。 |
| P0 | 完整一局无卡死 | 可发布 Demo 的底线。 |
| P0 | 终局能解释身份和分数 | 社交推理游戏必须能复盘。 |
| P1 | 配置表驱动 | 系统策划作品集的核心证据。 |
| P1 | 批量 AI 测试报告 | 平衡设计的核心证据。 |
| P1 | 新手引导 | 降低试玩门槛。 |
| P2 | 音效/动效 | 提升质感，但不决定规则验证。 |
| P2 | 角色变体 | 增加可玩性，但会放大平衡风险。 |

## 6. Demo 验收清单

| 检查项 | 标准 |
| --- | --- |
| 主流程 | 新开游戏 -> 身份确认 -> 7 回合 -> 讨论 -> 猜分 -> 结算，全程无异常。 |
| 文案 | 所有中文显示正常，无乱码、缺字、超框。 |
| UI | 主要分辨率下无重叠，按钮状态能区分可用/不可用。 |
| 规则 | 接触、拒绝、放弃、不可重复接触、连续不行动限制符合 PRD。 |
| AI | AI 不会选择非法目标，不会在无目标时卡住。 |
| 数据 | 批量测试 100 局无中断，统计数合计正确。 |
| 复盘 | 终局能展示身份、分数、关键事件和猜分结果。 |
| 构建 | Windows x64 包可独立运行。 |

## 7. 建议代码结构演进

当前原型可继续工作，但 Demo 前建议拆分职责：

| 当前 | 建议 | 收益 |
| --- | --- | --- |
| `GameManager` 同时管理规则、状态、AI 辅助 | `GameStateController` + `RuleResolver` + `SuspicionModel` | 降低规则修改风险。 |
| `UIManager` 驱动所有界面和 AI 协程 | `UIScreen` 子类或分面板 Controller | UI 更易维护。 |
| 玩家数据为普通 class | 保留运行时 PlayerState，配置另建 RoleConfig | 区分静态配置和动态状态。 |
| 直接拼接 UI 文案 | TextConfig 或本地化表 | 解决中文校对和后续多语言。 |
| 测试统计为 static 字段 | `SimulationReport` | 支持导出和版本对比。 |

## 8. 时间预算

| 目标版本 | 推荐周期 | 可交付程度 |
| --- | --- | --- |
| 快速 Demo | 5-7 天 | 修文案、加复盘、跑通构建，适合课堂/内部展示。 |
| 作品集 Demo | 10-14 天 | 配置表、批量测试、复盘 UI、平衡结论齐全，适合投递展示。 |
| 扩展 Demo | 3-4 周 | 加角色变体、规则编辑器、日志导出、音效动效。 |

## 9. 当前最值得立即做的三件事

1. 修复中文编码和所有 UI 文案，把“能看懂”作为第一目标。
2. 做 `GameEvent` 日志和终局复盘，把玩法推理链展示出来。
3. 把当前硬编码参数接入配置资产，和本目录 CSV 表一一对应。
